# Change: Post-Approval Profile Data Continuity (My Profile View & Edit)

## Why

Data captured during the 4-step membership registration wizard (Personal Information ŌåÆ Contact Information ŌåÆ Account Information ŌåÆ Additional Information) is not fully carried over to the member's **My Profile** page after admin approval. Members see empty or partial fields (e.g., Birthdate, Address, PRC License No., Company, photo) and are forced to re-enter data they already submitted. This creates data inconsistency between the application record and the member profile, and undermines trust in the approval workflow.

## What Changes

- **Registration and profile share a single member record.** Approval transitions the record's status (`Pending ŌåÆ Active`); it does not create or copy into a separate profile dataset.
- **My Profile auto-populates from registration data.** Every field submitted in steps 1ŌĆō4 appears in the corresponding section of My Profile immediately after approval, including uploaded files (profile photo, PRC ID).
- **My Profile gains explicit View and Edit modes, organized as 4 tabs mirroring the registration wizard's steps** ŌĆö Personal Information, Contact Information, Account Information, Additional Information. Each tab has its own View Mode (default, read-only) and its own Edit action/Save ŌĆö a member edits one section at a time, not one giant form. This applies once the application is submitted (`SubmittedAt` set), the same point the wizard already stops applying today ŌĆö not gated on approval.
  - **View Mode (default):** read-only display of that tab's fields plus system fields shown in the page header (Membership No., Status, Email, Member Type, Chapter).
  - **Edit Mode:** member can update that tab's self-service fields (e.g. address, company, contact details, photo, PRC License No.) with validation and a per-tab Save action.
- **The wizard gains two fields it was missing**, so "auto-populate from registration" is literally true for every field the profile shows: PRC License No. (text) joins step 1 "Personal Information" next to the existing PRC ID upload; Company becomes a real input in step 4 "Additional Information" (today just a review + terms screen).
- **System-controlled fields are protected.** Membership No., Status, Email, Member Type, and Chapter are all read-only in Edit Mode ŌĆö the member can view them but cannot change them; Member Type/Chapter changes are admin-managed only (see Decisions below).
- **Editing PRC License No. requires re-verification, staged as a real pending state** ŌĆö superseding the original "reset a flag to false" design. Changing the value requires re-uploading the PRC ID document in the same edit; the new value is staged as `PendingPrcLicenseNo` rather than overwriting `PrcLicenseNo` immediately, so the prior verified value keeps standing (and is what View Mode/the admin record shows) until an admin decides. Admins review from a dedicated **PRC Verifications** queue (mirrors the existing Membership Approvals queue/topbar notification pattern) and either **Approve** (the pending value becomes current, `PrcIdVerified` set `true`) or **Reject** (the pending value is discarded, a reason is recorded and shown to the member on their profile). Every decision is recorded in a `PrcVerificationHistory` audit trail (old value, new value, document reference, decision, reason, admin, timestamp). This same queue also covers **first-time verification** ŌĆö a member's PRC License No. entered at registration and never since reviewed, which previously had no admin-facing surface at all.
- **The raw "PRC ID Verified" checkbox in the admin Member edit form is removed.** Superseded by the Approve/Reject flow, so every verification decision goes through the audit trail rather than a bypassable toggle.
- **Saved edits propagate.** Changes are reflected immediately in View Mode and in the admin-side member record.

## Impact

- Affected specs: `member-profile` (new capability spec)
- Affected code (indicative):
  - `Domain`: Member aggregate ŌĆö `PrcIdVerified` (bool), `PendingPrcLicenseNo` (string?), `PrcVerificationRejectedReason` (string?); new `PrcVerificationHistory` entity (audit trail); registration fields already live on the Member entity (see Gap Analysis)
  - `Application`: `MemberService`/`UpdateMyProfileRequest` ŌĆö lock `MemberType`/`Chapter` to admin-only edits, require+consume a PRC ID re-upload and stage `PendingPrcLicenseNo` (not an immediate overwrite) when `PrcLicenseNo` changes; new `ApprovePrcVerificationAsync`/`RejectPrcVerificationAsync`; `GetAllAsync` gains a `pendingPrcVerificationOnly` filter
  - `Infrastructure`: EF Core migration for the new `Member` columns + `PrcVerificationHistory` table
  - `Web`: My Profile page (View/Edit toggle), form partials reusing the same field definitions as the registration wizard; new admin **PRC Verifications** page/queue + topbar notification section; `PrcVerificationRejectedReason` banner on the member's own profile
- No data migration/backfill needed: this codebase never split registration data into a separate application dataset ŌĆö see Gap Analysis. `tasks.md`'s original backfill tasks (1.4, 4.4) are cut as moot rather than implemented.

## Decisions

1. **Member Type and Chapter**: read-only for members, admin-managed only. While the application is still a draft (`SubmittedAt == null`, mid-wizard), both fields remain freely editable as part of completing the wizard. The instant the application is submitted, both fields lock permanently ŌĆö from then on they can only be changed by an admin via the existing `PUT /api/members/{id}` endpoint, never through the member's own Edit Mode, including via a crafted request. Approval does not change the lock state further; the lock is tied to submission, not approval.
2. **PRC License No.**: editing it requires re-uploading the PRC ID document in the same edit; the proposed value is staged (`PendingPrcLicenseNo`), not written directly, until an admin **Approves** (value becomes current, `PrcIdVerified` set `true`) or **Rejects** (pending value discarded, a reason is recorded and shown to the member). The edit is rejected if the value changes without an accompanying re-upload. This rule applies from submission onward, same boundary as Decision 1 ŌĆö during the initial wizard entry, no re-verification friction applies since nothing has been verified yet. A rejection shows an in-app banner on the member's Personal Information tab (no email system exists to notify any other way) ŌĆö it clears the next time the member attempts another PRC change.
3. **Email change flow**: out of scope. Not implemented as part of this change; email remains read-only everywhere it already is today.
4. **Post-submit UI structure**: My Profile is organized as 4 tabs/sections mirroring the wizard's steps (Personal/Contact/Account/Additional Information), each with an independent View(default)/Edit toggle and Save ŌĆö see "What Changes" above.
5. **Admin PRC verification queue**: a dedicated **PRC Verifications** page (mirrors Membership Approvals) lists every member with a pending change (`PendingPrcLicenseNo` set) or a never-yet-reviewed PRC License No. (`!PrcIdVerified`, nothing pending ŌĆö covers first-time verification, which previously had no admin-facing surface at all). Admins Approve or Reject from there; the raw "PRC ID Verified" checkbox on the generic admin Member edit form is **removed** so every decision goes through this queue and its audit trail (`PrcVerificationHistory`), with no bypass path.
